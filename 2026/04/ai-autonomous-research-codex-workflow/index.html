我现在怎么做科研：从文献调研、实验到论文写作

发表于2026-04-26|工程实践科研方法
|总字数:5.3k|阅读时长:17分钟

过去一段时间，我一直在反复调整自己的科研流程。

不是因为我想把所有事情都交给工具，而是因为很多时间其实都消耗在一些重复但又不能出错的环节里：查论文、配环境、跑实验、盯日志、补图表、整理初稿。

我现在比较稳定的一套做法，大致是这样的：

  • 用 ChatGPT Deep Research 做文献调研，产出带来源的报告、论文表格和 baseline 候选
  • 把调研结果整理成一页 experiment contract
  • 通过 本机代理 + 远程服务器 + 容器代理注入，让服务器里的 Codex 能稳定拉代码、装依赖、下模型、跑实验
  • 在容器或宿主机里装 Codex，让它先做 repo 勘察、再跑 smoke test、再执行 full run
  • 用 heartbeat 模式每隔几小时检查日志、GPU、checkpoint、指标和失败状态
  • 把实验结果整理成 run report + evidence table
  • 最后再交给我本机里的论文写作 skill，从 repo、结果和引用资产出发生成论文初稿

如果只看工具名，这篇文章会很容易写成一份“AI 工具推荐”。
但我更想讲清楚的是：这些工具到底怎么接到一起，哪些地方确实省事，哪些地方仍然必须自己盯。

一、真正耗时间的，不是科研本身，而是上下文切换

很多机器学习项目，真正拖慢你的，不是某个模型结构，而是大量碎片化、低杠杆但必须做对的工作：

  • 读论文时，方法谱系和 baseline 候选没有沉淀成实验计划
  • 复现时，环境依赖、CUDA、PyTorch、数据路径不断来回修
  • 跑实验时，日志、seed、config、checkpoint 没有统一记录
  • 长实验期间，没人盯着 loss、磁盘、OOM、NaN 和异常停止
  • 写论文时，Introduction 里的 claim 和 Experiments 里的证据又断开了

这些工作都不是“灵感”，但它们决定了科研项目最后能不能收束。

所以我现在会尽量把科研流程写成一串能接得上的步骤：

对我来说，工具真正有用的地方，不是帮我替代判断，而是帮我把这些步骤接稳。

二、我现在怎么用 Deep Research 做前期调研

我现在用 Deep Research，主要不是为了“快速看个综述”，而是为了给后面的实验和写作打底。

根据 OpenAI 官方帮助中心，Deep Research 现在支持：

  • 选择数据来源：公共网页、上传文件、连接的 apps
  • 在开始前先给出 research plan，你可以修改后再运行
  • 运行时实时跟踪进度，并中途打断调整方向
  • 输出结构化报告，并附 citations 或 source links
  • 结果可导出为 Markdown、Word、PDF

这几点对科研很重要，因为前期调研不能只停在“看起来总结得不错”，而要能回查来源，也要能继续往下走到实验设计。

我现在更常用的调研 prompt

我会明确告诉 Deep Research：这次调研不是为了写一篇泛综述，而是为了后续复现、设计 baseline 和写论文。

三、我会把调研结果写成一页实验说明

我越来越觉得，很多“工具没帮上忙”的情况，不是工具不行，而是你没有把问题说清楚。

如果你直接对 Codex 说：

那几乎等价于没说。

它不知道：

  • 要复现到什么程度
  • 哪些 baseline 是主线，哪些只是参考
  • 指标差多少算可接受
  • 先做 smoke test 还是直接 full run
  • 失败后能修环境到什么程度
  • 哪些事必须等你回来决策

所以我现在基本都会先写一个 experiment_contract.md，哪怕只是一页纸。

我会放进去的内容

这张 contract 的作用，其实很朴素：让后面的执行不要跑偏。

它真正解决的是两个问题：

  1. 防止 Codex 在开放空间里无限发散
  2. 防止“实验跑起来了，但其实跑偏了”

第二个问题尤其危险。
很多科研浪费，不是报错，而是表面上在跑，实际上协议早就漂了。

四、我怎么让服务器里的 Codex 真正帮我干活

我的实验并不是都在本机跑。更常见的模式是：

  • Mac 本机负责代理能力、调研、编辑、审阅
  • 远程 GPU 服务器负责训练、评测和长时间运行
  • Codex 安装在服务器容器或宿主机里，直接在实验环境旁边工作

我现在比较稳的一条网络链路

学校或实验室服务器，最容易卡人的不是算力，而是外部访问能力不稳定：GitHub、模型权重、Python 包、API、论文页面，任何一环都可能断。

五、长实验我会交给 heartbeat 去巡检

长实验最浪费人的，不是启动，而是等待过程中的不确定性。

比如这些情况，你大概率都遇到过：

  • loss 突然 NaN
  • dataloader 卡死
  • GPU 还占着，但训练进程已经挂了
  • checkpoint 没保存
  • 磁盘快满了
  • 验证指标长时间不动
  • 某个 seed 明显偏离，但没人知道该不该停

这时候 heartbeat 的作用，不是“帮你偶尔看一眼”，而是把观察、归类、低风险修复、日志沉淀做成一个定期小循环。

六、这套流程里我最在意的几个风险点

我最警惕的是下面这几种“伪成功”：

1. 跑起来了，但协议已经漂了

比如：

  • 数据预处理换了，但 baseline 还是按旧协议比
  • eval 脚本改了，但论文里的表格口径没更新
  • seed 少了一个，但你还在拿均值写结论

这时候系统不会报错，但科学上已经不可信。

2. loss 降了，但并不支撑论文 claim

有些实验只是说明“这个实现能学”，并不说明“这个方法更强”。
所以我现在会把论文里的强 claim 先写成一句话，再倒推它需要哪些证据。

3. 文本写得很顺，但引用其实没核过

这是我最不愿意放水的地方。

我本机里长期装着的 20-ml-paper-writing skill，有一条我非常认同的硬规则：
不要凭记忆生成 citation。BibTeX 必须程序化获取和验证。

这是对的。因为 citation 错误不是小瑕疵，而是学术风险。

所以我现在会把这条原则扩展到整套写作链：

  • citation 可以候选，但必须验证
  • 验证不了，就标 placeholder
  • 不能因为句子写顺了，就假装引用也对了

七、我会专门维护一张 evidence table

实验结果要真正变成论文，不靠记忆，也不靠聊天记录，而要靠一个明确的证据表。

我现在会维护类似这样的文件：

这个表的意义非常大。
它会直接约束你后面的写作：

  • Abstract 里的强 claim，必须能在 evidence table 里找到证据
  • Introduction 里的 contribution，必须能在 Experiments 对应到实验
  • 找不到证据的 claim，要么弱化，要么补实验，要么删掉

我觉得这一步特别重要，因为很多时候不是结果不够，而是写的时候不自觉把话说重了。
evidence table 的作用，就是在动笔之前先把证据链钉住。

八、写初稿时，我更相信 repo 和结果，而不是空写

我现在写论文初稿，不会让 AI 面对一张白纸，而是让它从这些东西里长出来：

我本机里会优先把这些 skill 串起来：

  • 20-ml-paper-writing
  • research-paper-writing
  • paper-figure-polish
  • systematic-debugging
  • verification-before-completion

它们分别解决的是不同层的问题：

  • 20-ml-paper-writing：从 research repo 出发生成论文初稿，尤其强调 citation 不能凭记忆生成
  • research-paper-writing：检查段落 message、claim-evidence 对齐、reviewer 视角下的逻辑强度
  • paper-figure-polish：在投稿前专门做图表与 caption 的质量收口
  • systematic-debugging：避免 AI 看到报错就乱修
  • verification-before-completion：避免它在没有 fresh evidence 的情况下说“已经好了”

如果项目更复杂，我还会加：

  • dispatching-parallel-agents
  • subagent-driven-development
  • executing-plans
  • jupyter-notebook
  • pdf

其中我尤其推荐一件事：
把实验命名、日志结构、写作模板、图表规范做成 repo 级 skill，而不是每次靠 prompt 重复。

九、图表我现在分成两类来处理

论文和公众号图，在我这里是两条完全不同的工作流。

第一类：数据图

像这些：

  • benchmark 柱状图
  • ablation table
  • convergence curve
  • robustness plot
  • sensitivity analysis

这类图必须从真实实验数据生成：

这部分不能交给图像模型“画个差不多的”。

第二类：概念图

像这些：

  • pipeline figure
  • teaser figure
  • 方法概念示意图
  • 公众号封面图

这类图，我现在更愿意先用 ChatGPT Images 快速生成草图，再做局部编辑和后续可编辑化。
OpenAI 官方帮助中心现在把 ChatGPT Images 明确描述成“create + edit existing images”的工作流，这对于概念图很方便：先把气质、构图和信息组织拉出来，再去做可编辑修正。

我之前也会用 NanoBananaPro2，再通过 EditBanana 转成可编辑图。
现在我的倾向是：

  1. 先用 ChatGPT Images 出第一版草图
  2. 再做局部编辑，确定构图和元素
  3. 最后落到 Figma / Illustrator / EditBanana 里做文字和矢量修正

简单说就是：

  • 数据图要可复现
  • 概念图先把草图做出来，再做成可编辑版本

十、另外几个我觉得确实有用的工具

如果你已经在用 ChatGPT + Codex，我觉得下面这些工具值得补上。

1. uv

如果你经常在服务器、容器和临时环境里切 Python 依赖，uv 很值得上。
Astral 官方文档把它定义成一个“extremely fast Python package and project manager”。对科研场景来说，它最大的意义不是“快一点”，而是更容易把环境初始化写成可重复脚本。

2. MLflow 或 W&B

如果你已经开始积累多个实验矩阵，只靠本地日志和截图很快就会乱。

  • 想要开源、自托管、把参数、代码版本、指标、输出文件系统化记录下来，MLflow 很合适
  • 想要更快做可视化比较、报告和团队协作，W&B 更顺手

MLflow 官方对 Tracking 的表述很直接：它就是用来记录 parameters、code versions、metrics 和 output files 的。
这跟科研真正需要的“实验账本”是同一个问题。

3. OpenAlex + Semantic Scholar API

如果你要做 citation 验证、paper table 维护、venue / author / works 元数据拉取，这两个非常值得接进自己的科研流水线。

  • OpenAlex 的 API 适合做 works / authors / sources 级别的结构化查询
  • Semantic Scholar API 适合做 paper / citation / venue / recommendations 相关能力

它们的价值不只是“找论文”，而是让你把引用、基线和 related work 变成结构化资产，而不是一堆手工复制的网址。

4. Snakemake

如果你的项目已经不只是 train.py 和 eval.py，而是开始出现：

  • 数据清洗
  • 特征生成
  • 多阶段训练
  • 批量评测
  • 表格 / 图像汇总

那就很值得用一个 workflow system 把它串起来。
Snakemake 对我来说最有用的地方，不是什么“工作流理念”，而是把多阶段实验流程真正固定下来。

十一、我优化后的整套 SOP

如果把这套流程压成一份最小可执行 SOP，我现在会这么做。

阶段 1：文献输入

  • 用 Deep Research 做一轮带来源的调研
  • 导出 paper_table.csv、baseline_matrix.csv
  • 标出可复现 baseline 和高风险 baseline

阶段 2：实验契约

  • 写 experiment_contract.md
  • 明确 smoke test / full run / stop rules
  • 明确 logging rules 和结果目录结构

阶段 3：执行层

  • 打通本机代理到远程服务器容器
  • 在容器里装 Codex
  • 让 Codex 先做 repo 勘察，不要直接改代码
  • 先跑 smoke test，再开 full run

阶段 4：观测层

  • 用 heartbeat 周期性读取 log / metrics / checkpoint
  • 状态只允许分成 running / failed / suspicious / converged / blocked
  • 只允许低风险修复和重启

阶段 5：证据层

  • 每个实验都写 run_report.md
  • 所有论文强 claim 都进 evidence_table.md
  • 找不到证据的 claim 一律降级处理

阶段 6：写作层

  • 用本机论文写作 skill 从 repo 和 evidence 出发写初稿
  • 引用先验证再进 bib
  • 图表先做数据图，再做概念图

这里最重要的其实不是某一个工具，而是所有中间状态都能留下来。

十二、几条我不会放松的原则

我并不相信“工具可以自动完成科学发现”这种说法。
至少在今天，我会给它明确的边界。

1. 研究问题还是得自己定

工具可以帮你扫 gap，但什么问题值得做，还是自己的判断。

2. 实验协议必须自己过一遍

工具可以建议 baseline 和指标，但公平性、对齐口径和数据泄漏风险不能全交给它。

3. 论文里的强 claim 必须自己负责

语言可以让 AI 起草，但结论责任不能外包。

4. citation 必须能回查

不能凭印象写引用，不能拿“像真的”当真的。

5. 给工具的权限要收紧

有些事情可以自动执行、自动观察、自动整理、自动起草；
但它不应该替你决定科研价值，也不应该无限制修改核心研究方向。

结语

我现在越来越觉得，做科研最怕的不是辛苦，而是流程散、上下文丢、很多关键决定最后没有留下痕迹。

在这套系统里：

  • Deep Research 负责外部知识输入
  • Codex 负责靠近实验环境的执行
  • heartbeat 负责长期观测和低风险恢复
  • evidence table 负责 claim 与结果对齐
  • 写作 skill 负责从 repo 中长出论文初稿
  • 图像能力负责概念图和可视化草稿
  • 人类负责问题、判断、边界和最终责任

我现在想要的，也不是“一键发论文”。
更现实一点说，我只是希望把那些重复、脆弱、容易丢上下文的工作整理顺一点，把时间尽量留给问题本身。
